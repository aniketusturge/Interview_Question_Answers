Cloud Infrastructure & DevOps Interview Q&A
Terraform & Infrastructure as Code
Q1: What is Terraform and how have you used it in your projects?
A: Terraform is an open-source Infrastructure as Code tool by HashiCorp that lets you define and provision cloud infrastructure using declarative configuration files. In my work, I've used Terraform to provision AWS resources like EC2 instances, VPCs, S3 buckets, and IAM roles. I maintain reusable modules, manage state files in S3 with DynamoDB locking, and integrate Terraform plans into CI/CD pipelines for automated deployments.

Q2: What is Terraform state and why is it important?
A: Terraform state is a file (terraform.tfstate) that maps your configuration to real-world resources. It tracks what infrastructure exists so Terraform knows what to create, update, or destroy. In team environments, we store state remotely — typically in S3 with state locking via DynamoDB — to prevent concurrent modifications and ensure consistency across team members.

Q3: How do you handle Terraform drift?
A: Terraform drift happens when real infrastructure diverges from the state file, usually due to manual changes. I handle it by running terraform plan regularly to detect differences, using terraform refresh to sync state with reality, and enforcing a policy that all changes go through Terraform — no manual console edits in production. For larger environments I've also used tools like Atlantis for automated plan/apply workflows.

AWS Cloud Services
Q4: Walk me through how you would secure an S3 bucket in AWS.
A: I would: block all public access using the S3 Block Public Access setting, enforce bucket policies to allow only specific IAM roles or services, enable server-side encryption (SSE-S3 or SSE-KMS), enable versioning and MFA Delete for critical data, turn on S3 access logging and CloudTrail for auditing, and use VPC endpoints so traffic doesn't traverse the public internet.

Q5: How do you manage IAM security in AWS?
A: I follow the principle of least privilege — users and roles get only the permissions they need. I use IAM roles for EC2 and Lambda instead of long-term access keys, enforce MFA for all human users, regularly audit permissions using IAM Access Analyzer and AWS Trusted Advisor, rotate credentials on a schedule, and avoid using the root account for day-to-day operations.

Q6: How would you troubleshoot an EC2 instance that is unreachable?
A: I follow a layered approach — first check if the instance is in a running state in the console, then review Security Group inbound rules and Network ACLs for the relevant port. Next I check the route tables and whether the subnet has an internet/NAT gateway. I'd review the instance's system log and EC2 serial console output for boot errors, verify the key pair, and check if the SSM agent is running as an alternative access method if SSH is blocked.

Cloud Security & Vulnerability Remediation
Q7: How do you approach VM vulnerability remediation in a cloud environment?
A: I start by triaging findings from a scanning tool like AWS Inspector or Qualys by severity — Critical and High first. For each finding I identify the affected resource, check if a patch is available, test in a non-prod environment, and apply via automated patching (SSM Patch Manager for AWS) or by rebuilding the instance from a hardened AMI. I track remediation SLAs, document the fix, and verify re-scan results to confirm closure.

Q8: What common cloud misconfigurations have you worked on remediating?
A: Common ones I've dealt with include: publicly exposed S3 buckets, overly permissive security groups (0.0.0.0/0 on port 22 or 3389), IAM users with admin policies attached directly, unencrypted EBS volumes or RDS snapshots, CloudTrail or logging disabled in regions, and EC2 instances running in public subnets without any need for it. I use AWS Config rules and Security Hub to continuously detect and alert on these.

CI/CD & Automation
Q9: Describe your experience with CI/CD pipelines for infrastructure.
A: I've worked with tools like GitHub Actions, Jenkins, and GitLab CI to automate Terraform workflows. A typical pipeline runs terraform fmt and terraform validate on every PR, then terraform plan for review, and terraform apply after approval on merge to main. For application deployments I've set up pipelines that build Docker images, push to ECR, and deploy to ECS or Kubernetes using rolling updates.

Q10: How have you used Python for cloud automation?
A: I've written Python scripts using Boto3 to automate tasks like stopping idle EC2 instances on a schedule, cleaning up unattached EBS volumes, sending SNS alerts when S3 bucket policies change, and generating cost reports from Cost Explorer. I've also written Lambda functions triggered by CloudWatch Events for automated remediation — like auto-revoking overly permissive security group rules.

GCP
Q11: What is your experience with GCP, and how does it compare to AWS?
A: My primary experience is in AWS, with basic exposure to GCP. Key parallels are: EC2 ↔ Compute Engine, S3 ↔ Cloud Storage, IAM works similarly though GCP uses a resource hierarchy with Org → Folder → Project. GCP's BigQuery is a standout service with no direct AWS equivalent in simplicity. I've worked with GCP for deploying VMs, managing GCS buckets, and setting up basic IAM roles and service accounts.

Incident Response & Operations
Q12: How do you handle a cloud security incident, for example an IAM key being exposed?
A: Immediately deactivate and delete the exposed key to stop further use. Then review CloudTrail logs to identify all API calls made with that key — look for unusual regions, services, or resource creation. Assess blast radius: were any resources created, data accessed, or permissions escalated? Notify the security team and follow the incident response runbook. Post-incident, enforce secrets scanning in CI/CD, move to IAM roles instead of keys, and document the timeline and remediation steps.

Scenario 1: Terraform State Corruption

"You run terraform apply and get an error saying the state file is locked or corrupted. Your team needs the deployment done urgently. What do you do?"

A:
First, I check if another pipeline or team member has a legitimate lock on the state — I'd look at the DynamoDB lock table for the lock entry and confirm with the team. If it's a stale lock from a crashed process, I use terraform force-unlock <lock-id> after confirming no active apply is running.
If the state file itself is corrupted, I:

Restore the last known good state from S3 versioning
Run terraform plan to verify it matches real infrastructure
If there's drift, use terraform import to re-import any missing resources

Going forward I'd ensure the pipeline always cleans up locks on failure and that S3 versioning is enabled on the state bucket.

AWS Config — Complete Guide
What is AWS Config?
AWS Config is a continuous monitoring and assessment service that records the configuration of your AWS resources over time and evaluates them against desired rules. Think of it as a CCTV camera for your AWS infrastructure — it constantly watches what your resources look like, tracks every change, and alerts you when something doesn't meet your standards.

Core Concepts
1. Configuration Items (CI)

Every time a resource changes, AWS Config captures a snapshot called a Configuration Item. It records:

Resource type and ID
Current configuration state
Relationships with other resources
Who made the change and when (via CloudTrail integration)
Tags
Think of it as a photograph of a resource at a specific point in time.

2. Configuration History

AWS Config stores the full history of all configuration items for every resource. This lets you answer questions like:

"What did this EC2 instance's security group look like 3 months ago?" "When was this S3 bucket made public and who did it?"

3. Configuration Snapshot

A point-in-time capture of all your AWS resources and their configurations delivered to an S3 bucket. Useful for audits and compliance reporting.

4. Config Rules

Rules are the heart of AWS Config. They define the desired state of your resources and continuously evaluate whether resources comply.

Two types:

Type	Description	Example
AWS Managed Rules	Pre-built by AWS — ready to use	s3-bucket-public-read-prohibited
Custom Rules	You write the logic using Lambda	Check if EC2 tags follow your naming convention
5. Conformance Packs

A collection of Config rules and remediation actions packaged together for a specific compliance framework like:

CIS AWS Foundations Benchmark
PCI DSS
HIPAA
SOC 2
Instead of adding 50 rules one by one, you deploy one conformance pack.

6. Remediation Actions

When a resource is found non-compliant, Config can automatically fix it using SSM Automation documents.

Example: If an S3 bucket is found public → automatically enable Block Public Access.

How AWS Config Works — Flow
Resource Changes
      ↓
AWS Config Recorder detects change
      ↓
Configuration Item created and stored in S3
      ↓
Config Rules evaluate the new state
      ↓
Resource marked Compliant or Non-Compliant
      ↓
SNS Notification / EventBridge Alert fired
      ↓
Optional: Auto-Remediation via SSM Automation

This is a very common interview question:

Feature	AWS Config	CloudTrail
Focus	What does a resource look like?	Who did what and when?
Tracks	Resource configuration state	API calls and user actions

Use case	Compliance, drift detection	Audit trail, forensics
Works best	Together — Config detects the change, CloudTrail shows who made it	Together with Config

  name     = "main-recorder"
  role_arn = aws_iam_role.config_role.arn

    all_supported                 = true
    include_global_resource_types = true

# Delivery channel — send to S3
resource "aws_config_delivery_channel" "main" {
  name           = "main-channel"
  s3_bucket_name = aws_s3_bucket.config_bucket.id
  depends_on     = [aws_config_configuration_recorder.main]
}

# Enable a managed rule — restrict SSH
resource "aws_config_config_rule" "restricted_ssh" {
  name = "restricted-ssh"

  source {
    owner             = "AWS"
    source_identifier = "INCOMING_SSH_DISABLED"
  }

  depends_on = [aws_config_configuration_recorder_status.main]
}

Interview Questions on AWS Config
Q: What is the difference between AWS Config and AWS Security Hub?

A: Config detects and records resource configuration changes and evaluates rules. Security Hub aggregates findings from Config, GuardDuty, Inspector, Macie, and other services into a single dashboard with a unified compliance score. Config feeds findings into Security Hub — they work together.

Q: Can AWS Config fix issues automatically?

A: Yes, through remediation actions using SSM Automation documents. You can set auto-remediation to trigger immediately when a rule is violated, with a configurable retry count.

Q: What happens if AWS Config is disabled in a region?

A: Config stops recording resource changes in that region. You lose visibility into configuration history and compliance status for all resources there. This is itself a compliance violation — which is why the cloud-trail-enabled and multi-region recording rules exist.

Q: How does AWS Config support multi-account environments?

A: Using AWS Config Aggregator, you can collect compliance data from multiple accounts and regions into a single central account. This is typically set up in an AWS Organizations management account to give a company-wide compliance view.
